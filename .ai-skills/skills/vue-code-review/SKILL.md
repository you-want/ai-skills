# 前端代码审查规范

## 概述

本 skill 用于检查前端代码（以 Vue 3 + TypeScript + SCSS Module 为主）是否符合团队规范，同时从 SOLID 原则、架构设计、代码质量、安全风险和可移除性等高级工程师视角进行审查。支持审查单个文件、多个文件、Git diff / 提交记录。

## 代码生成指引

当用户要求**生成、修改、重构**前端代码时，本 skill 同样适用。请在编写代码时**主动遵循**以下原则，确保输出代码即符合规范，无需二次审查修正：

### 文件与结构
- 使用 `<script setup lang="ts">`，组件顶部用 `defineOptions({ name: 'XxxComponent' })` 声明名称
- 样式使用 SCSS Module（`index.module.scss`），通过 `:class="style['class-name']"` 绑定
- 常量提取到同级 `constants.ts`，BO/Param 放到 `model/bo/` 和 `model/param/`
- 复杂逻辑优先提取为 composable，避免在组件中堆积

### TypeScript 规范
- **禁止使用 `any`**，必须定义具体类型
- Props 使用 `withDefaults(defineProps<PropsType>(), { ... })`，数组/对象默认值用箭头函数
- Emits 使用类型化定义（`const emit = defineEmits<{ (e: 'change', value: string): void }>()`），禁止字符串数组
- 响应式对象解构使用 `toRefs` 或 `storeToRefs`

### 模板与逻辑
- **模板中禁止写复杂业务逻辑**，所有计算和条件判断应在 `<script>` 中完成
- 标签属性顺序：条件（`v-if/v-show`）→ 样式（`class/style`）→ 属性（`prop`）→ 事件（`@event`）
- `v-for` 必须有 `key`，可变列表避免用 `index` 作为 `key`
- 每个组件根容器显式设置 `width` / `height`

### 命名与国际化
- 常量：UPPER_SNAKE_CASE；变量/函数：camelCase；类型/组件：PascalCase
- 硬编码中文文案替换为 `useI18n` 的 `t()` 方法

### 质量与性能
- 空值/undefined 必须做安全处理，条件判断使用 `===`
- 并行请求使用 `Promise.all`
- `watch` / 事件监听 / 定时器在 `onUnmounted` 中清理
- 避免重复代码，可复用逻辑提取为工具函数或 composable

### 安全与可维护性
- 禁止未经净化使用 `innerHTML` / `v-html`
- 禁止用户输入直接拼接 URL 或 DOM
- 避免魔法数字和硬编码字符串
- 修改接口/类型时，同步检查调用方、mock、测试是否需要更新

## 审查模式（每次审查前必须询问）

**每次执行审查前，必须先询问用户选择审查模式，不允许自动推断。**

### 询问话术

```
XQ

| 模式 | 审查范围 | 预计时间 | 适用场景 |
|------|---------|---------|---------|
| 快速 | 只审 P0 红线（12 组） | 30 秒 | 改文案、调样式、修 bug |
| 标准 | P0 全部 + P1 核心（24 组） | 2-3 分钟 | 新组件、新页面、常规开发 |
| 完整 | P0 + P1 + P2 全部（50 组） | 5-10 分钟 | MR/PR 合码前、架构调整、重构 |

请回复：快速 / 标准 / 完整
```

### 强制规则

- **不允许根据提交内容自动推断模式**
- **不允许根据文件类型自动推断模式**
- **不允许根据改动大小自动推断模式**
- 即使前一次审查用户选择了某模式，**下一次审查仍需重新询问**
- 用户明确说"快速审查" → 仍需确认："确认使用快速模式，只审 P0 红线？"
- 用户明确说"完整审查" → 仍需确认："确认使用完整模式，审查全部 50 组维度？"
- 用户只说"审查"或"review" → **必须询问**，不能直接开始审查

## 审查流程

1. **询问模式**：向用户展示模式选项，等待用户明确选择
2. **确认模式**：复述用户选择的模式及审查范围，获得确认
3. **理解上下文**：读取目标文件、diff、commit message，明确改动意图和作者
4. **按优先级逐项检查**：先扫 P0，再扫 P1（如模式包含），最后扫 P2（如模式包含）
5. **分级定级**：每个问题必须标注 `Block` / `Warning` / `Suggestion`
6. **输出报告**：按固定格式输出，包含通过项、问题项、改进建议、移除/迭代计划

## 优先级定义

审查时必须按 **P0 → P1 → P2** 的顺序执行，确保红线问题不被遗漏：

- **P0（Block）**：必须修复。合码阻塞项，包括安全漏洞、正确性 Bug、团队规范红线
- **P1（Warning）**：应该修复。潜在 Bug、可维护性问题、性能隐患
- **P2（Suggestion）**：建议优化。命名改进、minor 重构、缺少测试、注释补充

## 各模式审查范围

### 快速模式范围（P0 全部）

**必审维度：**
- P0-1 TypeScript 类型安全（any、Emits）
- P0-2 模板规范（复杂逻辑、异常调用）
- P0-3 组件基础规范（defineOptions、Props 默认值）
- P0-4 命名规范（常量/变量/类型命名）
- P0-5 样式规范（width/height、SCSS Module）
- P0-6 生命周期与资源清理（watch/事件/定时器清理）
- P0-7 可维护性（魔法数字）
- P0-8 输入输出安全（XSS、注入、SSRF、路径遍历、原型污染）
- P0-9 运行时异常防护（空值、数组越界、吞异常、错误处理）
- P0-10 认证与授权（租户检查、RBAC、IDOR）
- P0-11 密钥与隐私（硬编码凭证、PII 泄露）
- P0-12 密码学（弱算法、ECB 模式）

**跳过维度：** P1、P2 全部跳过，有问题也不输出（除非 P0 问题导致连锁反应）

### 标准模式范围（P0 全部 + P1 核心）

**P0：** 全部 12 组（同上）

**P1 核心维度（前 12 组）：**
- P1-1 文件结构（注释、script setup、constants、BO/Param 位置）
- P1-2 组件命名补充（XxxView/XxxPage/XxxDialog）
- P1-3 导入顺序（type → 第三方 → @/ → .vue → .scss）
- P1-4 TypeScript 类型补充（Props/Emits 类型、interface 定义）
- P1-5 Emits 补充（事件参数类型）
- P1-6 样式规范补充（class 绑定、内联样式）
- P1-7 暴露与双向绑定（defineModel、defineExpose）
- P1-8 模板规范补充（属性顺序、v-for key）
- P1-9 代码逻辑（空值处理、===、复杂条件、Promise、并行请求）
- P1-10 可维护性补充（未使用变量、重复代码、Prop 透传）
- P1-11 性能（虚拟滚动、computed 副作用、模板表达式、缓存 length、Promise.all）
- P1-12 国际化（useI18n、硬编码中文）

**跳过维度：** P1-13~P1-22（SOLID、代码异味、边界条件、JWT、竞态条件等深度维度）、P2 全部

### 完整模式范围（P0 + P1 + P2 全部）

**全部 50 组维度都审查**，包括：
- P0：12 组
- P1：22 组（含 SOLID、代码异味、边界条件、JWT、竞态条件、数据完整性等）
- P2：16 组（含测试、AI 风险、移除计划、可访问性等）

## 审查维度（按优先级分组）

### P0 — 红线规则（必须修复，合码阻塞）

#### P0-1. TypeScript 类型安全
- [ ] **禁止使用 `any` 类型** — 必须定义具体类型或 interface
- [ ] **Emits 必须使用类型化定义** — 禁止字符串数组 `defineEmits(['change'])`

#### P0-2. 模板规范
- [ ] **模板中禁止写复杂业务逻辑** — 所有计算和条件判断应在 `<script>` 中完成
- [ ] **模板中禁止直接调用可能抛异常的方法或访问深层属性**

#### P0-3. 组件基础规范
- [ ] **必须使用 `defineOptions({ name: 'ComponentName' })`** — 组件顶部声明名称
- [ ] **Props 必须为所有属性设置默认值** — 使用 `withDefaults(defineProps<PropsType>(), { ... })`
- [ ] **数组/对象 Props 默认值必须使用箭头函数**

#### P0-4. 命名规范
- [ ] **常量**：UPPER_SNAKE_CASE
- [ ] **变量/函数**：camelCase
- [ ] **类型/接口/组件**：PascalCase
- [ ] **BO/Param**：模块 [+ 动作/名词] + BO/Param
- [ ] **API**：动作 + 模块 + [具体数据] + [By条件]
- [ ] **枚举**：以 Type 或 Status 结尾

#### P0-5. 样式规范
- [ ] **每个组件根容器必须显式设置 width / height**
- [ ] **必须使用 SCSS Module**（`index.module.scss`）

#### P0-6. 生命周期与资源清理
- [ ] **watch / 事件监听 / 定时器必须在 `onUnmounted` 中清理** — 防止内存泄漏

#### P0-7. 可维护性
- [ ] **禁止魔法数字和硬编码字符串** — 必须提取为命名常量

#### P0-8. 输入输出安全
- [ ] **禁止 XSS** — 不安全的 HTML 注入、`dangerouslySetInnerHTML`、未转义的模板、`innerHTML` 赋值（Block）
- [ ] **禁止注入攻击** — 通过字符串拼接或模板字面量导致的 SQL/NoSQL/命令/GraphQL 注入
- [ ] **禁止 SSRF** — 用户控制的 URL 未经白名单校验就访问内部服务
- [ ] **禁止路径遍历** — 用户输入进入文件路径且未做清理（`../` 攻击）
- [ ] **禁止原型污染** — 不安全的对象合并（`Object.assign`、展开运算符与用户输入混用）
- [ ] **禁止用户输入直接用于 URL/DOM**（Block/Warning）
- [ ] **禁止敏感信息硬编码**（Warning）

#### P0-9. 运行时异常防护
- [ ] **禁止在热路径代码中留下未处理的潜在抛异常点**
  - 对可能为 `null` / `undefined` 的对象直接调用方法或访问深层属性（未使用可选链 `?.` 或前置判断）
  - 数组操作前未检查长度或有效性（如 `arr[0].x` 当 `arr` 可能为空）
  - 使用非空断言 `!` 或类型断言 `as` 掩盖潜在的空值风险
  - `JSON.parse`、`eval`、`new Function` 等未包裹 `try-catch`
  - 数字除法未对除数做零值检查
  - 直接调用第三方库方法时未验证参数有效性
- [ ] **禁止吞掉异常** — 空的 catch 块，或 catch 里只有日志
- [ ] **禁止过于宽泛的 catch** — 捕获 `Exception`/`Error` 基类而不是具体类型
- [ ] **禁止错误信息泄露** — 向用户暴露堆栈或内部细节
- [ ] **禁止缺失错误处理** — IO、网络、解析等易错操作没有 try-catch
- [ ] **禁止异步错误处理缺失** — 未处理的 Promise 拒绝、缺少 `.catch()`、没有错误边界

#### P0-10. 认证与授权
- [ ] **读写操作必须有租户或所有权检查**
- [ ] **新端点必须有认证守卫或 RBAC 强制校验**
- [ ] **禁止信任客户端提供的角色/标识/ID**
- [ ] **禁止越权访问（IDOR）** — 不安全的直接对象引用
- [ ] **禁止会话固定或弱会话管理**

#### P0-11. 密钥与隐私信息
- [ ] **禁止 API Key、Token、凭证硬编码在代码/配置/日志中**
- [ ] **禁止密钥出现在 Git 历史或暴露给客户端的环境变量中**
- [ ] **禁止过度日志记录 PII 或敏感载荷**
- [ ] **错误消息中必须有数据脱敏**

#### P0-12. 密码学
- [ ] **禁止弱算法** — MD5、SHA1 用于安全场景
- [ ] **禁止硬编码 IV 或盐值**
- [ ] **禁止无认证的加密** — ECB 模式、无 HMAC
- [ ] **禁止密钥长度不足**

### P1 — 规范规则（应该修复，影响质量）

#### P1-1. 文件结构
- [ ] 文件头部是否有组件功能描述注释
- [ ] 是否使用 `<script setup lang="ts">`
- [ ] 常量是否放在同级 `constants.ts` 中
- [ ] BO/Param 是否放在 `model/bo/` 和 `model/param/` 中

#### P1-2. 组件命名补充
- [ ] 组件名是否体现类型（XxxView / XxxPage / XxxDialog）

#### P1-3. 导入顺序
按以下顺序检查：
1. 类型导入（`import type`）
2. 第三方库
3. 项目内部模块（`@/` 开头）
4. 组件导入（`.vue` 文件）
5. 样式导入（`.module.scss`）

#### P1-4. TypeScript 类型补充
- [ ] Props / Emits 是否有具体类型定义
- [ ] 复杂类型是否单独定义 interface / type
- [ ] 非空断言 `!` 无守卫 — 建议用可选链或显式判断
- [ ] `as` 强制类型转换 — 可能隐藏真实问题

#### P1-5. Emits 补充
- [ ] 事件参数类型是否正确定义

#### P1-6. 样式规范补充
- [ ] 是否通过 `:class="style['class-name']"` 绑定
- [ ] 是否避免使用内联样式

#### P1-7. 暴露与双向绑定
- [ ] 是否使用 `defineModel`（Vue 3.4+）处理双向绑定
- [ ] 需要暴露给父组件时，是否使用 `defineExpose`

#### P1-8. 模板规范补充
- [ ] 标签属性顺序是否正确：条件 → 样式 → 属性 → 事件
- [ ] `v-for` 是否有 `key`，且避免用 index 作为 key（可变列表）

#### P1-9. 代码逻辑
- [ ] 空值/undefined 是否做了安全处理
- [ ] 条件判断是否使用了 `===`
- [ ] 复杂条件是否拆分为有意义的变量或函数
- [ ] 循环中是否有不必要的重复计算
- [ ] async/await 和 Promise 是否正确使用，错误是否处理
- [ ] 并行请求是否使用了 `Promise.all`
- [ ] 计算属性是否正确使用，避免在模板中写复杂表达式
- [ ] 响应式对象解构是否使用了 `toRefs`

#### P1-10. 可维护性补充
- [ ] 是否有未使用的变量或导入
- [ ] 复杂逻辑是否有注释说明
- [ ] 是否存在重复代码可提取复用
- [ ] 数组索引隐式耦合 — 依赖数组顺序而非 Map 关联
- [ ] 深层 Prop 透传 — 3 层以上建议用 provide/inject

#### P1-11. 性能
- [ ] 大列表是否考虑虚拟滚动或分页
- [ ] `computed` 中是否避免副作用
- [ ] 模板中是否避免复杂表达式
- [ ] 循环中是否缓存 `length`
- [ ] 并行请求是否使用 `Promise.all`
- [ ] 是否避免了不必要的重渲染或重复计算
- [ ] 异步操作是否有竞态条件处理
- [ ] 热路径中的昂贵操作：正则编译、JSON 解析、循环中的加密运算
- [ ] 不必要的重复计算：同一计算多次执行
- [ ] 缺少记忆化：纯函数被反复调用但结果未缓存
- [ ] N+1 查询：循环中逐条查询而不是批量查询
- [ ] 缺少索引：在没建索引的列上查询
- [ ] 过度获取：`SELECT *` 但只需要少量字段
- [ ] 没有分页：把整个数据集加载到内存
- [ ] 缺少缓存：重复 API 调用、DB 查询、计算
- [ ] 缓存没有 TTL：数据永久不过期
- [ ] 缓存没有失效策略：数据已更新但缓存未清理
- [ ] 无界集合：数组/Map 无限增长
- [ ] 大对象持有引用：阻止垃圾回收

#### P1-12. 国际化
- [ ] 是否使用 `useI18n` 的 `t()` 方法
- [ ] 硬编码中文文案是否应替换为国际化 key

#### P1-13. SOLID 原则
- [ ] **SRP（单一职责）**：文件是否承载了不相关的职责（如 HTTP + DB + 业务规则混在一起）
- [ ] **SRP**：类/模块是否过大、内聚性低、有多个变更理由
- [ ] **SRP**：函数是否编排了太多不相关的步骤
- [ ] **SRP**：是否出现上帝对象（知道系统太多细节）
- [ ] **OCP（开闭原则）**：新增行为是否需要修改大量 switch/if 代码块
- [ ] **OCP**：功能扩展是否需要修改核心逻辑，而不是通过扩展点实现
- [ ] **OCP**：是否缺少插件/策略/钩子机制
- [ ] **LSP（里氏替换）**：子类是否对具体类型做检查，或抛出基类方法不抛的异常
- [ ] **LSP**：重写方法是否弱化了前置条件或强化了后置条件
- [ ] **LSP**：子类是否忽略或空实现父类行为
- [ ] **ISP（接口隔离）**：接口是否包含很多方法，而实现者大多用不到
- [ ] **ISP**：调用方是否因为依赖宽泛接口而引入不必要的耦合
- [ ] **ISP**：是否出现空实现或桩实现
- [ ] **DIP（依赖倒置）**：高层业务逻辑是否直接依赖具体的 IO、存储或网络类型
- [ ] **DIP**：是否使用硬编码实现，而不是抽象或依赖注入
- [ ] **DIP**：导入链是否将业务逻辑与基础设施紧耦合

#### P1-14. 代码异味
- [ ] 过长方法：函数超过 30 行，嵌套层级过多
- [ ] 特性依恋：方法使用的数据更多来自其他类而非自身
- [ ] 数据团：同一组参数反复一起传递
- [ ] 基本类型偏执：用字符串/数字代替领域类型
- [ ] 霰弹式修改：一个改动需要修改很多文件
- [ ] 发散式变化：一个文件因很多不相关的原因被修改
- [ ] 死代码：永远不会执行的代码
- [ ] 投机泛化：为假设的未来需求提前做抽象

#### P1-15. 边界条件
- [ ] 缺少空值检查：在可能为 null 的对象上访问属性
- [ ] 真值/假值混淆：`if (value)` 在 `0` 或 `""` 是合法值时出错
- [ ] 空数组未处理：代码假设数组一定有元素
- [ ] 空对象边界：`for...in` 或 `Object.keys` 在空对象上的行为
- [ ] 首尾元素访问：`arr[0]` 或 `arr[arr.length-1]` 没有长度检查
- [ ] 除以零：除法前缺少检查
- [ ] 整数溢出：超出安全整数范围的大数
- [ ] 浮点数比较：使用 `===` 而不是误差范围比较
- [ ] 负值：本不应为负的索引或计数
- [ ] 越界错误：循环边界、数组切片、分页计算
- [ ] 空字符串未作为边界处理
- [ ] 仅空白字符的字符串：通过真值检查但实际为空
- [ ] 超长字符串：没有长度限制导致内存/显示问题
- [ ] Unicode 边界：表情符号、RTL 文本、组合字符

#### P1-16. JWT 与 Token 安全
- [ ] 算法混淆攻击（接受 `none` 或 `HS256` 但期望 `RS256`）
- [ ] 弱密钥或硬编码密钥
- [ ] 缺少过期时间（`exp`）或未校验
- [ ] JWT Payload 中存放敏感数据
- [ ] 未校验 `iss`（签发者）或 `aud`（受众）

#### P1-17. 运行时风险
- [ ] 无界循环、递归调用或超大内存缓冲区
- [ ] 外部调用缺少超时、重试或限流
- [ ] 在异步上下文中执行阻塞操作（同步 IO）
- [ ] 资源耗尽（文件句柄、连接数、内存）
- [ ] ReDoS（正则表达式拒绝服务）

#### P1-18. 竞态条件
- [ ] 共享状态并发访问：多线程/协程/异步任务在没有同步的情况下访问共享变量
- [ ] 全局状态或单例被并发修改
- [ ] 懒加载没有正确加锁（双检锁问题）
- [ ] Check-Then-Act（TOCTOU）：`if (exists) then use` 模式缺少原子操作
- [ ] `if (authorized) then perform` 中授权状态可能变化
- [ ] 先检查文件存在再操作文件
- [ ] 先检查余额再扣款（金融操作）
- [ ] 先检查库存再下单
- [ ] 数据库并发：缺少乐观锁（`version` 字段、`updated_at` 校验）
- [ ] 数据库并发：缺少悲观锁（`SELECT FOR UPDATE`）
- [ ] 读-改-写没有事务隔离
- [ ] 计数器自增没有原子操作（`UPDATE SET count = count + 1`）
- [ ] 并发插入导致唯一约束冲突
- [ ] 分布式系统：共享资源缺少分布式锁
- [ ] 分布式系统：缓存失效竞态（写入后读到旧数据）

#### P1-19. 数据完整性
- [ ] 缺少事务导致部分写入或状态不一致
- [ ] 持久化前校验过弱（类型强制转换问题）
- [ ] 可重试操作缺少幂等性设计
- [ ] 并发修改导致更新丢失

#### P1-20. 错误处理最佳实践
- [ ] 错误在合适的边界被捕获
- [ ] 错误提示对用户友好（不暴露内部细节）
- [ ] 错误日志包含足够的调试上下文
- [ ] 异步错误被正确传播或处理
- [ ] 可恢复错误有降级方案
- [ ] 关键错误触发告警/监控

#### P1-21. 供应链安全
- [ ] 依赖版本未固定，允许恶意更新
- [ ] 依赖混淆（私有包名与公共包冲突）
- [ ] 从未经验证的 CDN 导入资源且没有完整性校验
- [ ] 依赖存在已知 CVE 且长期未更新

#### P1-22. CORS 与响应头
- [ ] CORS 过于宽松（`Access-Control-Allow-Origin: *` 且允许凭据）
- [ ] 缺少安全响应头（CSP、X-Frame-Options、X-Content-Type-Options）
- [ ] 向外部暴露内部响应头或堆栈信息

### P2 — 建议规则（可选优化）

#### P2-1. 调试代码清理
- [ ] 代码中是否包含 `console.log`、`console.debug`、`console.table` 等调试输出
- [ ] 是否包含用于临时调试的 `debugger` 语句
- [ ] 是否包含被注释掉的废弃代码块或测试代码

#### P2-2. 可访问性（a11y）
- [ ] 图片是否有 `alt` 属性
- [ ] 表单元素是否有正确的 `label` 关联
- [ ] 弹窗/对话框是否有焦点管理（`autofocus`、`esc` 关闭）
- [ ] 颜色对比度是否可能过低（自定义样式覆盖 UI 组件时）

#### P2-3. 命名优化
- [ ] 变量名是否过于抽象（如 `list` → `userList`）
- [ ] 函数名是否准确描述行为

#### P2-4. 重构建议
- [ ] 复杂逻辑是否可提取为 composable
- [ ] 过长方法是否可拆分
- [ ] 是否可引入更合适的抽象

#### P2-5. 测试 — Bugfix 回归测试
- [ ] 修复 Bug 的 commit 是否包含对应的测试用例
- [ ] 测试是否覆盖了触发 Bug 的边界条件
- [ ] 测试是否验证了修复后的正确行为，而不仅仅是"不报错"

#### P2-6. 测试 — 复杂逻辑单元测试
- [ ] `computed` 中有复杂过滤/转换时，是否有独立测试
- [ ] 纯工具函数（utils/hooks）是否有单元测试
- [ ] 条件分支较多（>3 个）的函数是否覆盖了主要分支
- [ ] 数值计算、日期处理、字符串格式化是否有精确断言

#### P2-7. 测试 — 组件渲染与交互测试
- [ ] 关键用户路径是否有渲染测试
- [ ] Props 变化是否正确反映在 DOM 上
- [ ] 用户交互（点击、输入、提交）是否触发了正确的事件/回调
- [ ] 异步操作（加载、请求完成）是否有等待和状态断言
- [ ] 弹窗/对话框的打开关闭是否有测试

#### P2-8. 测试 — 测试与代码同步
- [ ] 修改了组件/函数后，相关测试是否同步更新
- [ ] 删除了功能后，对应的测试是否被清理（避免死测试）
- [ ] 重命名了文件/函数后，测试引用是否正确
- [ ] 改了类型定义后，测试中的 mock 数据是否仍然合法

#### P2-9. 测试 — 测试质量
- [ ] 测试名称是否清晰描述了测试场景
- [ ] 断言是否具体（避免 `toBeTruthy()` 代替精确值）
- [ ] 测试之间是否独立，没有隐式依赖执行顺序
- [ ] 是否避免了过于宽泛的 mock（mock 掉了整个模块导致测不到集成问题）
- [ ] 异步测试是否正确等待（`await`、`waitFor`、`flushPromises`）

#### P2-10. AI 生成代码特有风险 — 幻觉 API
- [ ] 调用的函数/方法是否在项目中真实存在
- [ ] 使用的组件属性是否在文档或类型定义中有声明
- [ ] 导入的模块路径是否正确（尤其 `@/` 别名、相对路径）
- [ ] 第三方库的 API 是否属于当前安装的版本

#### P2-11. AI 生成代码特有风险 — 上下文断章取义
- [ ] 修改了接口定义后，所有调用方是否同步更新
- [ ] 改了类型后，相关的 mock 数据、测试、storybook 是否同步
- [ ] 新增字段是否补充到了 BO/Param 类型中
- [ ] 是否遵循了项目已有的命名约定和目录结构
- [ ] 是否破坏了已有的设计模式（如把组合式逻辑写回了选项式）

#### P2-12. AI 生成代码特有风险 — 注释与代码不一致
- [ ] 注释描述的行为是否与代码实际行为一致
- [ ] 注释中提到的函数名、变量名是否与代码匹配
- [ ] TODO / FIXME 是否是真实遗留项，还是 AI 的占位符
- [ ] 注释是否过度解释显而易见的代码

#### P2-13. AI 生成代码特有风险 — 过度工程
- [ ] 是否引入了不必要的设计模式（工厂、策略、装饰器）
- [ ] 抽象层数是否过多，导致跟踪逻辑困难
- [ ] 是否为了"复用"而强行提取，反而增加耦合
- [ ] 简单的条件判断是否被过度泛化为配置驱动

#### P2-14. AI 生成代码特有风险 — 遗漏的连带修改
- [ ] 改了路由/URL 后，菜单配置、权限配置、面包屑是否同步
- [ ] 改了枚举值后，字典映射、下拉选项、显示文本是否同步
- [ ] 改了组件 Props 后，所有父组件调用处是否更新
- [ ] 改了常量后，所有引用处是否更新
- [ ] 改了国际化 key 后，所有语言包是否同步

#### P2-15. 可选链与类型
- [ ] 可选链过度使用：`a?.b?.c?.d` 掩盖了结构性问题
- [ ] null 与 undefined 混用：没有明确约定

#### P2-16. 移除与迭代计划
- [ ] 识别未使用、冗余、被特性开关关闭的代码
- [ ] 区分**现在可安全删除** vs **需要计划后移除**
- [ ] **P0**：立即移除（安全风险、显著成本、阻塞其他工作）
- [ ] **P1**：当前迭代移除
- [ ] **P2**：待办列表 / 下次迭代
- [ ] 提供后续步骤和验证检查点（测试/指标/监控）
- [ ] 搜索代码库中的所有引用（`rg`、`grep`）
- [ ] 检查动态/反射式使用
- [ ] 确认没有外部调用方（API、SDK、文档）
- [ ] 审查特性开关遥测数据（如适用）
- [ ] 更新/移除相关测试
- [ ] 更新文档
- [ ] 通知团队（如果是共享代码）

### G. 代码质量红线检查

审查时必须确认代码产出时就符合以下规范，而非依赖提交前修复：

#### G1. ESLint 风格合规性

- [ ] 引号：字符串使用单引号 `'`
- [ ] 分号：语句末尾不使用分号
- [ ] 缩进：2 个空格
- [ ] 未使用变量：已删除或改为 `_` 前缀
- [ ] 调试代码：`console.log`、`debugger` 是否已清理（**catch 块中的错误日志应使用 `console.error`**）
- [ ] 类型安全：无 `any` 类型
- [ ] 导入顺序：类型导入 → 第三方库 → 项目内部模块 → 组件导入 → 样式导入

#### G2. Commit Message 规范（如涉及提交信息）

- [ ] 格式是否符合 `type[(scope)]: subject`
- [ ] type 是否使用小写且属于允许枚举
- [ ] 冒号后是否有空格
- [ ] scope 为空时是否未写空括号 `()`

### H. 依赖变更审查

当 `package.json`、`pnpm-lock.yaml`、`yarn.lock`、`package-lock.json` 有变化时：

- [ ] 新增依赖是否必要，是否有更轻量的替代方案
- [ ] 依赖版本升级是否有 Breaking Changes
- [ ] 是否有已知 CVE 的依赖被引入或升级
- [ ] 开发依赖与生产依赖是否分类正确

## 针对 Git 提交记录的额外检查

- 通过 `git log` 和 `git diff` 获取改动内容
- 关联提交作者信息（只显示名称，不显示邮箱）
- 按提交分组输出审查结果
- `package.json` 版本升级视为正常，但大版本跳跃需提示确认 Breaking Changes
- 如果多个 commit 修改了同一类问题，合并说明

## 输出格式

必须按以下格式输出审查报告：

```markdown
## 代码审查报告

### 文件/提交：xxx/index.vue
### 作者：代码作者名称

#### ✅ 通过项

- P0: 无 any 类型使用
- P0: Emits 类型化定义
- P1: 使用 SCSS Module
- ...

#### ❌ 问题项

| 行号 | 优先级 | 严重等级 | 问题描述 | 规范要求 | 责任人 |
| ---- | ------ | -------- | -------- | -------- | ------ |
| 15 | P0 | Block | 使用了 any 类型 | 禁止使用 any，需定义具体类型 | 张三 |
| 23 | P0 | Block | 缺少 defineOptions | 组件顶部必须声明名称 | 张三 |
| 45 | P1 | Warning | 未清理 watch | watch 必须在 onUnmounted 中清理 | 张三 |
| 67 | P2 | Suggestion | 变量名 `list` 过于抽象 | 使用具体命名如 `userList` | 张三 |

#### 💡 改进建议

1. 建议将 `any[]` 改为 `DataItem[]`
2. 建议将 `defineEmits(['change'])` 改为类型化定义
3. ...
```

如果审查多个文件或多个提交，每个文件/提交单独输出一个报告块，最后加一个总体汇总和移除/迭代计划：

```markdown
## 总体汇总

| 项目/文件 | P0 Block | P1 Warning | P2 Suggestion | 整体评估 | 责任人 |
| --------- | -------- | ---------- | ------------- | -------- | ------ |
| xxx/index.vue | 1 | 2 | 1 | 需修改后合并 | 张三 |
| yyy.ts | 0 | 1 | 0 | 基本可接受 | 李四 |

## 移除/迭代计划
（如适用，列出可移除代码和后续计划）
```

## 责任人字段说明

- 从 Git 提交记录中获取作者名称
- 只填写代码作者名称，**不需要邮箱**
- 如果无法确定作者，填写 "未知"

## 重要约束

- **默认只审查，不修改代码**。除非用户明确要求实现修改，否则只输出审查报告。
- 如果审查通过且没有问题，明确说明检查了哪些维度，以及是否有未覆盖的区域。
- **审查时必须按 P0 → P1 → P2 顺序执行**，先确保红线问题无遗漏，再深入细节。
