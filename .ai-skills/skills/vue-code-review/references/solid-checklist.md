# SOLID 原则与代码异味检查清单

## SRP（单一职责原则）

- 文件是否承载了不相关的职责（如 HTTP + DB + 业务规则混在一起）
- 类/模块是否过大、内聚性低、有多个变更理由
- 函数是否编排了太多不相关的步骤
- 是否出现上帝对象（知道系统太多细节）
- **自问**："这个模块唯一会因为什么而改变？"

## OCP（开闭原则）

- 新增行为是否需要修改大量 switch/if 代码块
- 功能扩展是否需要修改核心逻辑，而不是通过扩展点实现
- 是否缺少插件/策略/钩子机制
- **自问**："我能否在不触碰现有代码的情况下添加新变体？"

## LSP（里氏替换原则）

- 子类是否对具体类型做检查，或抛出基类方法不抛的异常
- 重写方法是否弱化了前置条件或强化了后置条件
- 子类是否忽略或空实现父类行为
- **自问**："我能否在调用方不知情的情况下替换任何子类？"

## ISP（接口隔离原则）

- 接口是否包含很多方法，而实现者大多用不到
- 调用方是否因为依赖宽泛接口而引入不必要的耦合
- 是否出现空实现或桩实现
- **自问**："所有实现者都会用到这个接口的所有方法吗？"

## DIP（依赖倒置原则）

- 高层业务逻辑是否直接依赖具体的 IO、存储或网络类型
- 是否使用硬编码实现，而不是抽象或依赖注入
- 导入链是否将业务逻辑与基础设施紧耦合
- **自问**："我能否在不改变业务逻辑的情况下替换这个实现？"

## 常见代码异味

| 异味 | 特征 |
|------|------|
| **过长方法** | 函数超过 30 行，嵌套层级过多 |
| **特性依恋** | 方法使用的数据更多来自其他类而非自身 |
| **数据团** | 同一组参数反复一起传递 |
| **基本类型偏执** | 用字符串/数字代替领域类型 |
| **霰弹式修改** | 一个改动需要修改很多文件 |
| **发散式变化** | 一个文件因很多不相关的原因被修改 |
| **死代码** | 永远不会执行的代码 |
| **投机泛化** | 为假设的未来需求提前做抽象 |
| **魔法数字/字符串** | 没有命名常量的硬编码值 |

## 重构启发式原则

1. **按职责拆分，而不是按大小拆分** — 小文件也可能违反 SRP
2. **需要时才引入抽象** — 等到第二个用例出现再做抽象
3. **保持重构增量式** — 先隔离行为，再移动
4. **先保证行为不变** — 重构前先加测试
5. **按意图命名** — 命名困难说明抽象可能有问题
6. **优先组合而非继承** — 继承造成紧耦合
7. **让非法状态无法表示** — 用类型强制约束不变量
