# AI 生成代码特有风险检查清单

## 幻觉 API

AI 经常生成"看起来合理但实际上不存在"的 API 调用。

### 检查点

- [ ] 调用的函数/方法是否在项目中真实存在
- [ ] 使用的组件属性是否在文档或类型定义中有声明
- [ ] 导入的模块路径是否正确（尤其 `@/` 别名、相对路径）
- [ ] 第三方库的 API 是否属于当前安装的版本

### 常见模式

```ts
// ❌ 危险：假设存在的方法
someApi.fetchUserDetailById(id)  // 实际可能是 fetchUserInfoById

// ❌ 危险：假设组件有某个 prop
<ExpressionEditor append-event-name="xxx" />  // 需确认该 prop 真实存在且被消费
```

## 上下文断章取义

AI 只看到了局部代码，可能破坏全局约定。

### 检查点

- [ ] 修改了接口定义后，所有调用方是否同步更新
- [ ] 改了类型后，相关的 mock 数据、测试、storybook 是否同步
- [ ] 新增字段是否补充到了 BO/Param 类型中
- [ ] 是否遵循了项目已有的命名约定和目录结构
- [ ] 是否破坏了已有的设计模式（如把组合式逻辑写回了选项式）

### 常见模式

```ts
// ❌ 危险：改了返回结构但调用方没改
// 服务端接口增加了分页包装
// 但前端旧代码还在直接读取 res.list
```

---

## 注释与代码不一致

AI 写的注释往往描述的是"它想生成的理想代码"，而不是实际代码。

### 检查点

- [ ] 注释描述的行为是否与代码实际行为一致
- [ ] 注释中提到的函数名、变量名是否与代码匹配
- [ ] TODO / FIXME 是否是真实遗留项，还是 AI 的占位符
- [ ] 注释是否过度解释显而易见的代码

### 常见模式

```ts
// ❌ 危险：注释说"按 ID 查询"，实际是按名称查询
// 按用户 ID 获取详情
function fetchUserByName(name: string) { ... }
```

---

## 过度工程

AI 倾向于使用"更高级"的模式，有时反而增加复杂度。

### 检查点

- [ ] 是否引入了不必要的设计模式（工厂、策略、装饰器）
- [ ] 抽象层数是否过多，导致跟踪逻辑困难
- [ ] 是否为了"复用"而强行提取，反而增加耦合
- [ ] 简单的条件判断是否被过度泛化为配置驱动

### 常见模式

```ts
// ❌ 过度：一个简单表单引入了抽象工厂
// ❌ 过度：两个地方相似就强行提取，结果参数爆炸
```

---

## 遗漏的连带修改

AI 经常只改它看到的部分，忽略隐式依赖。

### 检查点

- [ ] 改了路由/URL 后，菜单配置、权限配置、面包屑是否同步
- [ ] 改了枚举值后，字典映射、下拉选项、显示文本是否同步
- [ ] 改了组件 Props 后，所有父组件调用处是否更新
- [ ] 改了常量后，所有引用处是否更新
- [ ] 改了国际化 key 后，所有语言包是否同步

### 常见模式

```ts
// ❌ 危险：改了 USER_STATUS 枚举，但页面上还有硬编码的 1/0
// ❌ 危险：改了 API 路径，但 mock 数据还在用旧路径
```

---

## 自问清单

- "这段代码是 AI 生成的吗？如果是，我需要额外验证哪些点？"
- "这个 API/属性/方法我在项目中真的见过吗？"
- "改了这里，还有哪些文件应该一起改？"
- "注释是在解释代码，还是在美化代码？"
- "这个抽象是必要的，还是为了看起来高级？"
