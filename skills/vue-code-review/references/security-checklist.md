# 安全与可靠性检查清单

## 输入输出安全

- **XSS**：不安全的 HTML 注入、`dangerouslySetInnerHTML`、未转义的模板、`innerHTML` 赋值
- **注入攻击**：通过字符串拼接或模板字面量导致的 SQL/NoSQL/命令/GraphQL 注入
- **SSRF**：用户控制的 URL 未经白名单校验就访问内部服务
- **路径遍历**：用户输入进入文件路径且未做清理（`../` 攻击）
- **原型污染**：不安全的对象合并（`Object.assign`、展开运算符与用户输入混用）

## 认证与授权

- 读写操作缺少租户或所有权检查
- 新端点没有认证守卫或 RBAC 强制校验
- 信任客户端提供的角色/标识/ID
- 越权访问（IDOR - 不安全的直接对象引用）
- 会话固定或弱会话管理

## JWT 与 Token 安全

- 算法混淆攻击（接受 `none` 或 `HS256` 但期望 `RS256`）
- 弱密钥或硬编码密钥
- 缺少过期时间（`exp`）或未校验
- JWT Payload 中存放敏感数据（Token 只是 base64，不是加密）
- 未校验 `iss`（签发者）或 `aud`（受众）

## 密钥与隐私信息

- API Key、Token、凭证硬编码在代码/配置/日志中
- 密钥出现在 Git 历史或暴露给客户端的环境变量中
- 过度日志记录 PII 或敏感载荷
- 错误消息中缺少数据脱敏

## 供应链与依赖

- 依赖版本未固定，允许恶意更新
- 依赖混淆（私有包名与公共包冲突）
- 从未经验证的 CDN 导入资源且没有完整性校验
- 依赖存在已知 CVE 且长期未更新

## CORS 与响应头

- CORS 过于宽松（`Access-Control-Allow-Origin: *` 且允许凭据）
- 缺少安全响应头（CSP、X-Frame-Options、X-Content-Type-Options）
- 向外部暴露内部响应头或堆栈信息

## 运行时风险

- 无界循环、递归调用或超大内存缓冲区
- 外部调用缺少超时、重试或限流
- 在异步上下文中执行阻塞操作（同步 IO）
- 资源耗尽（文件句柄、连接数、内存）
- ReDoS（正则表达式拒绝服务）

## 密码学

- 使用弱算法（MD5、SHA1 用于安全场景）
- 硬编码 IV 或盐值
- 使用无认证的加密（ECB 模式、无 HMAC）
- 密钥长度不足

## 竞态条件

竞态条件会导致间歇性故障和安全漏洞，需特别关注：

### 共享状态访问

- 多线程/协程/异步任务在没有同步的情况下访问共享变量
- 全局状态或单例被并发修改
- 懒加载没有正确加锁（双检锁问题）
- 并发场景下使用非线程安全集合

### Check-Then-Act（TOCTOU）

- `if (exists) then use` 模式缺少原子操作
- `if (authorized) then perform` 中授权状态可能变化
- 先检查文件存在再操作文件
- 先检查余额再扣款（金融操作）
- 先检查库存再下单

### 数据库并发

- 缺少乐观锁（`version` 字段、`updated_at` 校验）
- 缺少悲观锁（`SELECT FOR UPDATE`）
- 读-改-写没有事务隔离
- 计数器自增没有原子操作（`UPDATE SET count = count + 1`）
- 并发插入导致唯一约束冲突

### 分布式系统

- 共享资源缺少分布式锁
- 领导者选举竞态
- 缓存失效竞态（写入后读到旧数据）
- 事件顺序依赖但没有正确序列化
- 集群操作中的脑裂场景

### 常见危险模式

```
# 危险模式：
if not exists(key):
    create(key)

value = get(key)          # 读-改-写
value += 1
set(key, value)

if user.balance >= amount:  # Check-then-act
    user.balance -= amount
```

### 自问

- "如果两个请求同时命中这段代码会怎样？"
- "这个操作是原子的吗，还是可能被中断？"
- "这段代码访问了哪些共享状态？"
- "高并发下这段代码的行为会怎样？"

## 数据完整性

- 缺少事务导致部分写入或状态不一致
- 持久化前校验过弱（类型强制转换问题）
- 可重试操作缺少幂等性设计
- 并发修改导致更新丢失
